# MotionBERT Realtime 性能天花板判断

## 1. 结论摘要

基于当前工作区的 realtime 实现、近期多轮调度/传输优化结果，以及用户现场观感反馈，可以给出当前阶段的判断：

- 当前链路已经明显脱离“纯传输拖慢”的阶段，进入“推理主导”的阶段。
- 在不替换 MotionBERT 路线、不更换 AlphaPose 前端的前提下，当前稳定版已经接近这台机器的实时上限。
- 继续优化仍有空间，但收益会明显变小，且越来越接近“用复杂工程换有限提升”。
- 如果目标是继续显著降低延迟或再提升明显帧感，后续高收益方向将更多集中在：
  - 更短实时窗口
  - 更激进的推理节奏控制
  - 更轻或更适合实时的 2D 前端
  - 更成熟的一体化 UE/实时人体方案

## 2. 当前观测事实

来自用户现场反馈的关键观测：

- 当前稳定版姿态已经正确
- 相比更早版本，帧率与延迟已经明显改善
- 当前 CPU 占用约 `70%`
- 当前 GPU 占用 `90%+`

这组读数的含义非常重要：

- `GPU 90%+` 说明主压力已经落在视觉推理链路上，而不是还卡在简单的数据传输层
- `CPU 70%` 说明调度、可视化、UE 收包、Python 处理也在吃资源，但还不是唯一主瓶颈
- 如果是“传输没优化好”导致的慢，通常不会稳定把 GPU 顶到这个水平

## 3. 当前稳定版包含的有效优化

当前稳定版不是最原始版本，而是已经叠加了以下有效优化的版本：

1. UE 发送改为紧凑包
2. `positions` 语义修正为局部骨骼位移
3. UE 插件接收线程只消费最新包，不回放积压历史包
4. `--no_display` 时跳过无意义可视化开销
5. 主入口保留为用户体感更优的稳定版

因此，当前性能表现已经是“修掉明显低级瓶颈之后”的结果，不是原始基线。

## 4. 主要瓶颈判断

### 4.1 第一瓶颈：AlphaPose 2D 阶段

当前路线仍然是：

- 摄像头/视频输入
- AlphaPose 检测 + 2D pose
- MotionBERT 3D 推理
- UE Live Link 发送

AlphaPose 本身是重型两阶段方案，默认又使用 `yolox-x` 级别检测器。  
在当前链路中，它仍然大概率是最重的一段之一。

它的特征是：

- 单帧开销大
- 检测与 2D pose 叠加
- 对“端到端低延迟”不是天然最优解

### 4.2 第二瓶颈：MotionBERT 长时序窗口

当前 realtime 默认仍是较长的时序输入逻辑。  
这意味着随着运行时间增长，MotionBERT 会持续处理更长的历史序列，而最终只取最后一帧 3D 结果。

这会带来两个现实代价：

- 吞吐压力上升
- 响应速度下降

它不会直接破坏姿态语义，但会持续消耗 GPU 预算。

### 4.3 第三层瓶颈：剩余调度/展示开销

这部分已经被明显削弱，但不是完全为零：

- Python 线程调度
- OpenCV 帧转换/拷贝
- UE 编辑器预览开销
- Live Link 消费与显示层

这部分通常是“能继续抠一点”，但不太可能再带来数量级改善。

## 5. 为什么说已经接近当前机器上限

判断“接近上限”不是看有没有 100% 吃满全部资源，而是看：

1. 主瓶颈是否已经落在高成本模型推理上
2. 低级瓶颈是否已经被修掉
3. 再往下优化是否需要更高工程复杂度

当前这条链路满足这三个条件：

- GPU 已长期高占用
- 传输层已经做过压缩和最新包优先
- 错误语义的优化已经被修正
- 实验性并行流水线已尝试过，但用户体感反而更慢

所以现在更像是：

> 系统已经从“工程调度瓶颈”过渡到“模型推理瓶颈”

## 6. 后续优化收益预期

### 6.1 仍值得继续做的方向

#### A. 更短实时窗口

这是后续最值得继续实验的方向之一。

预期收益：

- 降低 MotionBERT 单次推理成本
- 降低端到端延迟
- 提升响应灵敏度

潜在代价：

- 动作平滑性或稳定性可能略降
- 某些复杂动作的 3D 质量可能下降

是否值得：

- 值得
- 这是“不换路线前提下”最像高收益的下一刀

#### B. 更激进的 inference cadence 策略

例如：

- 专门的低延迟档
- 更小 `min_frames`
- 更短 `clip_len`
- 更明确的实时模式参数组合

预期收益：

- 在不改语义的前提下继续压榨实时表现

是否值得：

- 值得
- 但收益预计会小于前一阶段的传输/语义修正收益

### 6.2 可以做，但预期收益有限的方向

#### A. 继续抠 UDP / JSON / 线程微优化

这类优化仍可能带来一点收益，但因为当前主瓶颈已经更多在推理侧，所以边际收益较小。

是否值得：

- 仅在非常明确发现了新的低级瓶颈时再做

#### B. 继续尝试更复杂的 Python 多线程流水线

已经试过一次更激进的流水线实验，用户体感反而更慢。  
这说明“结构更复杂”不等于“更快”。

是否值得：

- 当前阶段不建议作为第一优先级

### 6.3 若要继续明显变快，需要改变路线的方向

如果后续目标是：

- 延迟继续明显下降
- 帧感继续明显上升
- 仍然保持较高姿态质量

那么更有可能产生大收益的方向将是：

- 更适合实时的 2D 前端
- 更成熟的一体化实时人体方案
- UE 侧更原生/更成熟的实时人体接入路线

这就不再是“当前小修小补”的范围了，而是路线级优化。

## 7. 当前建议

### 建议 1：把当前稳定版作为可用基线固定

当前建议默认继续使用：

- `run_motionbert_realtime.bat`

原因：

- 用户体感已经验证过
- 姿态正确
- 延迟与帧感相比早期版本明显改善
- 版本管理已经建立完成

### 建议 2：后续只做两类实验

#### 实验类型 A：低延迟参数实验

范围：

- 更短窗口
- 更低启动帧门槛
- 更明确的低延迟参数预设

目标：

- 在不换路线前提下再挖一点收益

#### 实验类型 B：路线级替代方案调研

范围：

- 更适合实时的人体前端
- 更成熟的 UE 一体化方案

目标：

- 判断是否值得从“当前链路继续抠”转向“更适合实时的方案”

## 8. 最终判断

一句话总结：

> 当前稳定版已经是这条 MotionBERT realtime -> UE 路线在本机上的高可用版本，而且大概率已经接近“不换路线时”的现实性能天花板。

如果还要继续提升：

- 小提升：继续做低延迟参数实验
- 大提升：要考虑路线级调整
